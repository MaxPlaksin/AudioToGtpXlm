# Правила написания кода для Cursor AI

## 1. ЧИТАЕМОСТЬ И ПОНЯТНОСТЬ

### 1.1 Имена переменных и функций
- Используй осмысленные имена, которые объясняют назначение
- Избегай сокращений (кроме общепринятых: i, j для счетчиков)
- Функции называй с глаголами: `getUserData()`, `calculateTotal()`
- Булевы переменные начинай с is/has/should: `isActive`, `hasPermission`

### 1.2 Комментарии
- Комментируй ПОЧЕМУ, а не ЧТО (код сам должен объяснять ЧТО)
- Добавляй docstring для всех публичных функций
- Объясняй сложные алгоритмы и бизнес-логику
- Помечай TODO для незавершенной работы

### 1.3 Форматирование
- Следуй стандартному стилю языка (PEP 8 для Python)
- Используй последовательные отступы (4 пробела для Python)
- Оставляй пустые строки для группировки логических блоков
- Максимальная длина строки: 88-100 символов

## 2. ФУНКЦИОНАЛЬНОСТЬ

### 2.1 Принцип единственной ответственности
- Одна функция — одна задача
- Если функция делает больше одного действия — разбей её
- Максимальный размер функции: 20-30 строк

### 2.2 Чистота функций
- Функции должны быть детерминированными (одинаковый вход = одинаковый выход)
- Избегай побочных эффектов
- Предпочитай иммутабельность мутабельности

### 2.3 Обработка ошибок
- Всегда обрабатывай потенциальные ошибки
- Используй специфичные типы исключений
- Не глотай исключения без обработки
- Логируй ошибки с контекстом

## 3. ТЕСТИРОВАНИЕ

### 3.1 Покрытие тестами
- Пиши тесты для критической логики
- Тестируй граничные случаи
- Используй понятные названия тестов
- Один тест — одна проверка

### 3.2 Типы тестов
- Unit-тесты для отдельных функций
- Integration-тесты для взаимодействия компонентов
- E2E-тесты для критических пользовательских сценариев

## 4. БЕЗОПАСНОСТЬ

### 4.1 Валидация входных данных
- Всегда проверяй типы и форматы
- Экранируй спецсимволы
- Не доверяй пользовательскому вводу

### 4.2 Секреты и конфиденциальность
- Никогда не хардкодь пароли и ключи
- Используй переменные окружения
- Не логируй чувствительные данные

## 5. ПРОИЗВОДИТЕЛЬНОСТЬ

### 5.1 Эффективность
- Избегай ненужных вычислений в циклах
- Используй соответствующие структуры данных
- Кэшируй результаты тяжелых операций
- Ленивые вычисления где возможно

### 5.2 Ресурсы
- Закрывай соединения с БД
- Освобождай файловые дескрипторы
- Используй контекстные менеджеры

## 6. АРХИТЕКТУРА

### 6.1 Разделение ответственности
- Разделяй логику на слои (презентация, бизнес, данные)
- Используй dependency injection
- Избегай циклических зависимостей

### 6.2 Расширяемость
- Пиши код, который легко модифицировать
- Используй интерфейсы и абстракции
- Следуй принципам SOLID

## 7. КОНТРОЛЬ ВЕРСИЙ

### 7.1 Коммиты
- Делай маленькие, атомарные коммиты
- Пиши понятные сообщения коммитов
- Не коммить закомментированный код
- Не коммить секреты и конфиги

## 8. ДОКУМЕНТАЦИЯ

### 8.1 Код как документация
- Пиши самодокументируемый код
- Обновляй документацию при изменении кода
- Документируй архитектурные решения
